Mbrain

Once you're running eight AI agents, the agents stop being the hard part. Knowing what they're all doing becomes the hard part. Mbrain is the answer to that.

The problem

I build a lot of agents. A nonprofit's marketing fleet. A research pipeline. A coach. A set of native apps. Individually each one works. Collectively they became a fog — I couldn't tell you which were running, which had quietly died, which token was about to expire and take a publishing pipeline down with it.

Every agent had its own conventions, its own credentials, its own idea of where it kept state. The fleet had grown past what I could hold in my head, which is exactly the point at which most side-project sprawl starts rotting.

The operating loop

  1. Register

    Every agent gets one canonical record in agents.json: what it is, where it lives, how it runs, what port it answers on.

    Registry

  2. Probe

    Status detection asks reality, not the config — live port and process checks decide what counts as "running".

    status.py

  3. Monitor

    Every morning at 09:00 the credential monitor re-asks each provider — Meta, LinkedIn, Google — how long its token actually has left.

    launchd · daily

  4. Alert

    Anything within seven days of expiry triggers an email before it becomes an outage. Silent token death is the failure mode this kills.

    mailer.py

  5. Publish

    Gmail, LinkedIn, and Instagram are wired once, centrally — any agent can post through them instead of re-solving auth per project.

    Connectors

How Claude is used

  • operatorClaude Code is the operator console. It reads the registry, runs the probes, checks credential runway, and posts through the connectors — the whole fleet is drivable from a conversation.
  • memoryPersistent memory across sessions: Claude remembers how the fleet is wired, what changed, and what to check first — so operating it doesn't restart from zero each time.
  • builderEvery subsystem — registry, monitor, connectors, dashboard, and this site — was built in Claude Code sessions against the live system.
  • metaThis page is itself an Mbrain artefact: a Claude session probed the live fleet, read the real token expiry dates, and wrote what you're reading.

The idea underneath

The credential monitor doesn't read a stored expiry date and trust it — it asks each provider directly, every morning. Stored state drifts, and the failure mode of a silently-expired token is a publishing pipeline that looks healthy and posts nothing.

That's the whole philosophy in miniature. Probe reality, don't trust the record. It's the same instinct that governance work in healthcare teaches you: the system that says it's fine is not the same as the system that is fine.
